# Scénario simple de démonstration — SNA

## Nom du projet
**SNA — Suivi des Notes Académiques**

## Idée du projet
Le projet sert à gérer des notes d'étudiants de manière simple :
- ajouter une note pour un cours ;
- consulter les notes déjà encodées ;
- modifier ou supprimer une note ;
- calculer une moyenne générale ;
- préparer ensuite un petit tableau de bord.

L'idée est volontairement simple, logique et crédible pour une présentation étudiante.

## Comptes de démonstration
Mot de passe pour tous les comptes : **Demo1234!**

- **admin** : compte principal pour la présentation
- **po** : product owner
- **sm** : scrum master
- **dev** : développeur
- **test** : testeur
- **viewer** : lecture seule

## Ce que montre la base de données de démo
Le JSON contient :
- 1 projet nommé **SNA - Suivi des Notes Académiques**
- 3 sprints
- plusieurs tickets avec une hiérarchie simple :
  - des **epics**
  - des **user stories**
  - des **tasks**
  - des **bugs**
- des commentaires
- des activités

## Organisation du scénario
### Sprint 1 — Connexion
Le premier sprint est terminé.
Il sert à montrer que le projet a commencé par quelque chose de simple :
- connexion ;
- formulaire ;
- vérification des identifiants ;
- correction d'un bug.

### Sprint 2 — Gestion des notes
C'est le **sprint actif** pour la démonstration.
C'est celui qu'il faut montrer en priorité.

Dans ce sprint, il y a déjà des tickets répartis dans le Kanban :
- **To Do** :
  - Modifier ou supprimer une note
  - Bug sur la validation des coefficients
- **In Progress** :
  - Ajouter une note par cours
  - Calculer la moyenne après ajout
- **Done** :
  - Créer le formulaire d'ajout de note
  - Consulter la liste des notes
  - Afficher le tableau des notes

Ça permet de montrer directement :
- le board Kanban ;
- l'état d'avancement ;
- les colonnes ;
- les assignations ;
- les story points ;
- le burndown chart.

### Sprint 3 — Tableau de bord
Le troisième sprint est planifié.
Il sert à montrer qu'il y a déjà une suite logique du projet :
- afficher la moyenne générale ;
- préparer un export PDF.

## Déroulement conseillé pour la présentation
### 1. Connexion
Connecte-toi avec **admin**.

### 2. Liste des projets
Montre qu'il y a un projet clair et cohérent :
**SNA - Suivi des Notes Académiques**.

### 3. Product backlog
Montre que le backlog est organisé de manière logique :
- une partie connexion déjà terminée ;
- une partie gestion des notes en cours ;
- une partie tableau de bord prévue ensuite.

### 4. Sprint actif
Ouvre le **Sprint 2 - Gestion des notes**.
C'est la partie la plus importante pour la démo.

Tu peux dire par exemple :
> Dans mon projet, j'ai choisi un cas simple de gestion des notes.  
> J'ai déjà terminé la partie connexion, je suis en train de finaliser la gestion des notes,  
> et le tableau de bord est prévu pour le sprint suivant.

### 5. Kanban
Montre qu'il y a des tickets dans plusieurs colonnes.
C'est utile pour justifier :
- l'avancement du sprint ;
- le suivi des tâches ;
- le travail de l'équipe ;
- le côté visuel du projet.

### 6. Détail d'un ticket
Ouvre par exemple :
- **Ajouter une note par cours**
- **Modifier ou supprimer une note**
- **Consulter la liste des notes**

Tu pourras montrer :
- le type du ticket ;
- la priorité ;
- l'assignation ;
- les commentaires ;
- la relation entre les tickets.

### 7. Rapport / graphique
Ouvre le rapport du projet pour montrer le **burndown chart**.
Comme il y a déjà des tickets terminés dans le sprint actif, le graphique est plus crédible.

### 8. Rôles
Tu peux finir par montrer qu'il existe plusieurs comptes :
- un admin ;
- un scrum master ;
- un développeur ;
- un testeur ;
- un viewer en lecture seule.

## Pourquoi ce scénario est crédible
Ce scénario reste simple et naturel :
- il n'y a qu'un seul projet ;
- le sujet est facile à comprendre ;
- les tickets ont des noms réalistes ;
- le sprint actif contient déjà du contenu visuel ;
- on a l'impression que le projet a été construit progressivement.
